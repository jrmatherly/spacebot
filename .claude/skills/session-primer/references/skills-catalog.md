# Skills Catalog

The 22 project-level skills in `.claude/skills/`, organized by function. Each skill is invoked as a slash command (e.g., `/spacebot-dev`).

## Tier 1: Core Architecture Skills

These encode deep domain knowledge. Activate when working on the relevant subsystem.

### spacebot-dev (662 lines)
**Trigger:** Any Spacebot feature, bug fix, refactor, or architectural question. Any Rust code changes in `src/`, prompt template edits, migration work, tool additions.

**Covers:**
- Five process types (Channel, Branch, Worker, Compactor, Cortex) — roles, tool sets, max turns, isolation boundaries
- Event system — 23 ProcessEvent variants, broadcast buses (`event_tx`, `memory_event_tx`)
- Memory system — SQLite + LanceDB + redb, 8 memory types, 6 relation types, hybrid search (RRF ranking)
- Task system — lifecycle, TaskStatus enum, cortex ready-task loop
- Tool system — 60+ tools in 5 ToolServer configurations, adding new tools checklist
- LLM integration — model types, hook system, prompt caching, context windows
- Type aliases — AgentId, ChannelId, WorkerId, BranchId, ProcessType, ProcessId
- Configuration — hot-reload via ArcSwap, settings merging
- Security/sandbox — tool validation, Wasm runtime
- API server — HTTP endpoints, message serialization

### spacedrive-dev (352 lines)
**Trigger:** Spacedrive integration, the vendored `spacedrive/` subtree at the repo root, P2P networking, file indexing, sync systems, Spacebot-Spacedrive contract.

**Covers:**
- VDFS architecture — headless daemon + clients, CQRS pattern
- Data model — SdPath, Entry, ContentIdentity, Tag graph, Location, Device
- File indexing — 5 phases, 3 index modes, 8,500 files/sec target
- Sync — leaderless P2P, HLC, device-owned vs shared resources
- Networking — Iroh transport (QUIC), TLS 1.3, NAT traversal
- Extension system — WASM, 5 building blocks
- React integration — query hooks, semantic color system
- Spacebot integration — connection modes, HTTP + SSE, ContextNode, Archive

### spaceui-dev (247 lines)
**Trigger:** UI components, styling with SpaceUI tokens, AI chat interfaces, task management UI, integrating SpaceUI.

**Covers:**
- 6-package monorepo — tokens, primitives (41+ components), forms (8 wrappers), ai (15 components), explorer (4 components), icons (198 PNGs)
- Design tokens — semantic colors, 7 themes, Tailwind v4 CSS-first
- Component patterns — Button (8 variants), Dialog, Select, Toast, etc.
- AI components — ToolCall, MessageBubble, ChatComposer, TaskRow, TaskDetail
- Integration — Vite (local HMR), Next.js, React Native/NativeWind
- Conventions — bun only, TypeScript strict, forwardRef, CVA for variants

## Tier 2: OpenSpec Workflow Skills

The OpenSpec system manages structured change proposals through a lifecycle: explore → propose → implement → verify → archive. Changes live in `openspec/changes/`.

### openspec-explore (288 lines)
**Trigger:** "explore an idea", "investigate a problem", "think through", "clarify requirements"
- Read-only thinking mode — no implementation
- Can create OpenSpec artifacts to capture conclusions
- Useful before committing to a design direction

### openspec-propose (110 lines)
**Trigger:** "propose a change", "create a proposal", "new feature proposal"
- Creates proposal, design, specs, and task artifacts in one step
- Generates all OpenSpec change artifacts from a description

### openspec-apply-change (156 lines)
**Trigger:** "implement this change", "start implementing", "continue implementing"
- Reads context files, tracks progress, handles blockers
- Follows the task list generated by propose

### openspec-verify-change (168 lines)
**Trigger:** "verify the implementation", "check if it matches the spec"
- Three-dimensional verification: completeness, correctness, coherence
- Checks task completion, spec coverage, design adherence

### openspec-archive-change (114 lines)
**Trigger:** "archive this change", "finalize the change"
- Moves completed changes to `openspec/changes/archive/YYYY-MM-DD-<name>/`
- Checks artifact completion and task status

## Tier 3: Development Workflow Skills

### session-sync (166 lines)
**Trigger:** "sync", "update docs", "session done", "wrap up", after completing significant work
- Rebuilds code-review graph
- Checks CLAUDE.md for drift
- Syncs all 5 Serena memories against ground truth
- Commits doc fixes with convention: `docs: session sync — <description>`

### commit-all (36 lines)
**Trigger:** "commit all", "commit everything"
- Groups unrelated changes into separate commits
- Never uses `git add -A` or `git add .`

### pr-gates (44 lines)
**Trigger:** "open a PR", "prepare for review", "push this"
- Mandatory: `just preflight` → `just gate-pr` → push
- If same command fails twice, stop and debug root cause
- Review closure: code reference + verification command + pass/fail evidence per finding

### pr-slicer (55 lines)
**Trigger:** "split this PR", "make this smaller"
- Target ≤400 lines, ≤10 files, 1-4 commits per slice
- Order: prerequisites → refactors → behavior → UI/docs/polish

### review-fix-loop (47 lines)
**Trigger:** "address review feedback", "fix PR comments"
- Parse findings into P1/P2/P3
- Build closure table, implement smallest fix batch
- Run targeted verification, then broader gate checks

### deps-update (156 lines)
**Trigger:** "update dependencies", "check for updates", "audit deps"
- Two modes: safe (semver-compatible) and upgrade (breaking changes)
- Covers Rust (`cargo update/audit`), frontend (`bun update/audit`), docs
- Known constraints: protobuf pinned, fumadocs version guard

**Trigger:** "review the prompt", "audit the system prompt"

### docs-audit (398 lines)
**Trigger:** "audit docs", "review documentation", "find stale docs", "cleanup docs", "what's out of date"
- Systematic audit of Tier 1 (user-facing) and Tier 2 (operational) documentation against git history, codebase, and release artifacts
- Produces prioritized findings (🔴 Incorrect / 🟡 Stale / 🔵 Missing / ⚪ Polish) with file:line citations and evidence
- Scope excludes CLAUDE.md and Serena memories (owned by `session-sync`), OpenSpec change artifacts (owned by `openspec-verify-change`)

### session-primer
**Trigger:** Start of every new session, "prime", "bootstrap", "get up to speed", "new session"
- Bootstrap sequence: activate Serena project context, retrieve cross-session CMEM observations, assess git/working-tree state
- Loads 5 canonical memories, identifies task domain, recommends next skill to invoke
- Runs automatically via `/session-primer` at session start

## Skill Composition Patterns

Skills are frequently composed in sequence:

| Workflow | Skills Used |
|----------|-------------|
| **New feature** | `openspec-explore` → `openspec-propose` → `spacebot-dev` + `openspec-apply-change` → `openspec-verify-change` → `pr-gates` → `openspec-archive-change` |
| **Bug fix** | `spacebot-dev` → fix → `pr-gates` → `session-sync` |
| **Dependency update** | `deps-update` → `pr-gates` → `session-sync` |
| **Code review** | `review-fix-loop` → `pr-gates` |
| **Session end** | `session-sync` → `commit-all` (if needed) |
| **Deployment** | `cluster-context` → `cluster-deploy` → `cluster-ops` |
| **Parallel work** | `archon` (isolated worktrees) |
