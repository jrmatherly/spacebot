# Config Reference

Every option accepted by Spacebot's TOML configuration schema, with defaults, types, and examples. Companion to [Configuration](/docs/config) which covers how config loads, resolves, and hot-reloads.

**Source of truth:** `src/config/toml_schema.rs` in the spacebot crate. If a field exists in the schema but not on this page, the page is stale. File an issue or open a PR.

## How to Read This Page

Every field row has a type annotation. Common types:

| Type | Meaning |
|---|---|
| `bool` | `true` / `false` |
| `u8`, `u16`, `u32`, `u64`, `usize` | Non-negative integers |
| `i64` | Signed integer |
| `f32`, `f64` | Floating-point |
| `string` | UTF-8 string. May use `secret:NAME`, `env:VAR`, or literal. |
| `Option<T>` | Field may be omitted |
| `Vec<T>` | TOML array |
| `HashMap<K, V>` | TOML table |
| `UUID` | RFC 4122 UUID string |
| `PathBuf` | Filesystem path |

**Defaults shown in the "Default" column.** A default of `—` means the field is truly optional with no implicit value. Omitting it disables the feature.

## Value Resolution

Any string field supports three resolution modes:

```toml
api_key = "secret:ANTHROPIC_API_KEY"    # Secret store lookup (recommended)
api_key = "env:ANTHROPIC_API_KEY"       # Process environment
api_key = "sk-ant-..."                  # Literal (not recommended for credentials)
```

The secret store is the preferred path. See [Secret Store](/docs/secrets) for the category system and output scrubbing.

## `[llm.providers.<id>]` — Structured providers (table form)

Use these blocks to define providers that need more than an API key: Azure deployments, LiteLLM proxies, custom base URLs, Bearer auth, or extra headers.

| Field | Type | Default | Notes |
|---|---|---|---|
| `api_type` | `string` | required | One of: `openai_completions`, `openai_chat_completions`, `kilo_gateway`, `openai_responses`, `anthropic`, `gemini`, `azure`. See pairing rule below. |
| `base_url` | `string` | required | API endpoint. See pairing rule below. |

**`api_type` ↔ `base_url` pairing rule.** The `openai_*` variants differ in how they extend `base_url`:

| `api_type` | Appends | `base_url` must |
|---|---|---|
| `openai_completions` | `/v1/chat/completions` | NOT end in `/v1` (e.g., `http://host:4000`) |
| `openai_chat_completions` | `/chat/completions` | already include `/v1` (e.g., `http://host:4000/v1`) |
| `openai_responses` | `/v1/responses` | NOT end in `/v1` |
| `anthropic`, `gemini`, `azure`, `kilo_gateway` | provider-specific full paths | see provider examples below |

Mixing up the pairing yields a double-`/v1` path that silently 404s on the proxy side. If unsure, strip `/v1` from your `base_url` and use `openai_completions`.
| `api_key` | `string` | required | Credential (use `secret:` / `env:`) |
| `name` | `Option<string>` | — | Routing key override; defaults to `<id>` from the section header |
| `api_version` | `Option<string>` | — | Azure OpenAI api-version |
| `deployment` | `Option<string>` | — | Azure OpenAI deployment name |
| `extra_headers` | `Vec<(string, string)>` | `[]` | Custom HTTP headers forwarded on every request |
| `use_bearer_auth` | `bool` | `false` | Send as `Authorization: Bearer <key>` instead of provider default |

> **Singular form compat alias.** `[llm.provider.<id>]` and
> `[llm.providers.<id>]` are both recognized; the deserializer treats
> them identically. New configs should use the plural form for
> consistency with the schema and the array form `[[providers]]`. The
> singular form remains supported indefinitely for backward compatibility
> with configs written before 2026-04-20.

```toml
[llm.providers.anthropic]
api_type = "anthropic"
base_url = "https://api.anthropic.com"
api_key = "secret:ANTHROPIC_API_KEY"

[llm.providers.azure]
api_type = "azure"
base_url = "https://YOUR-RESOURCE.openai.azure.com"
api_key = "secret:AZURE_OPENAI_API_KEY"
api_version = "2025-04-01-preview"
deployment = "gpt-5-nano"

[llm.providers.litellm]
api_type = "openai_completions"                    # appends /v1/chat/completions to base_url
base_url = "http://localhost:4000"                 # NO /v1 here — the api_type appends it
api_key = "secret:LITELLM_API_KEY"
use_bearer_auth = true
extra_headers = [
    ["x-litellm-tags", "spacebot-prod"],
]
```

## `[[providers]]` — Array form (cluster-friendly)

Same fields as `[llm.providers.<id>]` but with a **required** `name` field. Useful when generating config from Kubernetes ConfigMaps or other templating systems that prefer array syntax.

| Field | Type | Default | Notes |
|---|---|---|---|
| `name` | `string` | required | Routing key. No `/`, no whitespace, non-empty. |
| `api_type` | `string` | required | Same values as table form |
| `base_url` | `string` | required | |
| `api_key` | `string` | required | |
| `api_version` | `Option<string>` | — | |
| `deployment` | `Option<string>` | — | |
| `extra_headers` | `Vec<(string, string)>` | `[]` | |
| `use_bearer_auth` | `bool` | `false` | |

**Conflict resolution:** if both `[llm.providers.X]` and `[[providers]] name = "X"` exist, the table form wins.
