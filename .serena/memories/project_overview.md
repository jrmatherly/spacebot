# Spacebot - Project Overview

**Version:** 0.5.1
**Language:** Rust (edition 2024), ~130K lines of code
**Purpose:** An agentic system where every LLM process has a dedicated role. Replaces the monolithic session model with specialized processes (Channel, Branch, Worker, Compactor, Cortex).

## Tech Stack
- **Runtime:** Tokio async
- **HTTP Framework:** Axum 0.8
- **Database:** SQLite via sqlx 0.8 (53 migrations total: 41 in flat per-agent `migrations/`, 12 in instance-wide `migrations/global/` including 5 added by Phase 2 on 2026-04-21, 2026-02 → 2026-04)
- **Vector DB:** LanceDB 0.27 (embedded vector + FTS for memory)
- **Key-Value Store:** redb 4.1 (embedded; bumped from 4.0 in PR #101 deps roll-up)
- **LLM Framework:** Rig v0.35 (agentic loop framework). LiteLLM proxy is a first-class provider as of 2026-04-20 (PR #78; UI Update/Test/Delete modal completed in PR #80 with baseUrl, use_bearer_auth, and extra_headers fields; Azure singular [llm.provider.azure] → plural [llm.providers.azure] migration landed in the same PR with read-fallback for legacy configs): `[llm.providers.litellm]` block with `api_type = "openai_completions"` transport (no new `ApiType` variant), `litellm_api_key` field on `LlmConfig` (System-category, virtual API key — NOT the proxy's admin master key), `ProviderStatus.litellm` bool for UI detection, `litellm/`-prefixed model names skip Spacebot's rate-limit tracking because the LiteLLM Router owns those semantics.
- **CLI:** clap 4.6 (bumped from 4.5 in PR #101)
- **Serialization:** serde/serde_json 1.0
- **Logging:** tracing 0.1
- **Error Handling:** thiserror + anyhow
- **Discord:** Serenity (rev-pinned from `next` branch at `1cbceb275b10566145b0bdca1c57da9502079a6a` — pulls rustls 0.23 + tokio-tungstenite 0.28 to resolve rustls-webpki audit advisories. Rev-pin replaced branch-tracking 2026-04-19 to stop `cargo update` re-fetches from invalidating incremental cache.)
- **WebSocket:** tokio-tungstenite 0.28
- **Metrics:** Prometheus (feature-gated behind `metrics` feature)
- **macOS Keychain:** security-framework 3
- **Embeddings:** fastembed **exact-pinned to 5.13.2**. 5.13.3 pulled hf-hub 0.5.0 which broke ONNX model fetch across 8 memory tests. Relax the pin once hf-hub 0.5 compatibility is restored upstream.

## Security

- Dependabot alerts in spacebot-owned code that are blocked on upstream crate updates are tracked in-repo at `docs/security/deferred-advisories.md` (GHSA, severity, blocker, unblock trigger)
- Non-dismissal policy: deferred advisories stay `open` on the Security dashboard; do NOT dismiss via the GitHub API
- Spacedrive-scoped Dependabot alerts (under `spacedrive/**`) remain open until the planned runtime integration; re-triaged at integration time
- `.github/dependabot.yml` covers all shipped-code manifests (root cargo, desktop/src-tauri, interface, packages/api-client, spaceui and its sub-manifests, docs); it controls update-PR scoping only, not security-alert visibility
- **Entra ID auth Phase 0 landed 2026-04-20 (PR #81, commit `2a40131`)**: constant-time bearer compare via `subtle = "2"` crate (closes obs #25356); JWT-shape regex added to `LEAK_PATTERNS` at `src/secrets/scrub.rs`; `spacebot_auth_failures_total{branch, reason}` Prometheus counter backed by a typed `AuthRejectReason` enum with 4 reasons (`header_missing`, `header_non_ascii`, `scheme_missing`, `token_mismatch`) and an exhaustiveness-guarded label-snapshot test that compile-fails if a variant is added without a corresponding assertion; CORS-credentials regression test (with positive `access-control-allow-origin` assertion proving the CorsLayer is engaged); `ApiState::new_for_tests` + `api::test_support::build_test_router` always-compiled `#[doc(hidden)]` test scaffolding. Static `auth_token` path preserved unchanged.
- **Phase 1 landed 2026-04-20 (PR #82, squash commit `91b8e38`)**: new `src/auth/` module tree (`errors`, `context`, `config`, `jwks`, `middleware`) validates Entra v2 JWTs via `jwt-authorizer 0.15` with RS256 pinned from `jsonwebtoken 9`, distinguishes User (delegated, has `scp`) vs ServicePrincipal (app-only, no `scp`), enforces required-scope for users plus required-role for SPs, and emits the `entra_jwt` metric branch. `ApiState.entra_auth: Arc<ArcSwap<Option<Arc<EntraValidator>>>>` follows the `task_store`/`wiki_store` setter pattern. Branch selection in `start_http_server` picks Entra when configured, static-token otherwise (never both attached). Both branches now populate `AuthContext` into request extensions. Precursor: existing `src/auth.rs` (Anthropic OAuth PKCE) renamed to `src/anthropic_oauth.rs` to free the `auth` namespace (10 call sites migrated, git-preserved). Test coverage: 5 unit (`auth::context`, `auth::config`) + 16 integration (`tests/entra_jwt_middleware.rs`: 11 validator-level + 5 router-level via `build_test_router_entra`) + Wiremock-backed `tests/support/mock_entra.rs` with pure-Rust `rsa 0.9.10` keypair generation. Config surface: `[api.auth.entra]` TOML block with `env:`/`secret:` resolution via new `resolve_required_ref` helper that fails-fast on unresolvable references. Design record: `docs/design-docs/entra-app-registrations.md`.
- **Phase 2 landed 2026-04-21 (PR #101, squash commit `bd7579b`)**: authz data model + fire-and-forget user upsert. Five additive instance-level migrations (`users`, `teams`, `team_memberships`, `service_accounts`, `resource_ownership` — the last is a sidecar keyed on `(resource_type, resource_id)` per research §12 Alternative-1 rather than sprouting owner/visibility columns on 20 per-agent tables). Identity key is composite `{tid}:{oid}` surfaced as `principal_key`. CHECK constraints lock every authz enum column (`visibility`, `principal_type`, `users.status`, `teams.status`, `source`) to its allowed value set. New `src/auth/principals.rs` (record types + `Visibility` enum) and `src/auth/repository.rs` (CRUD helpers) target the instance pool. `upsert_user_from_auth` returns a typed `RepositoryError` enum (`InvalidPrincipalType` for legacy-static rejection, `Sqlx(#[from] sqlx::Error)` otherwise); the other three helpers return `anyhow::Result` with `.with_context()`. `ApiState.instance_pool: ArcSwap<Option<sqlx::SqlitePool>>` mirrors the setter pattern. Middleware fires-and-forgets the upsert on successful Entra auth, wrapped in `tracing::info_span!("auth.upsert_user", principal_key=...).instrument(...)`. Two new Prometheus counters: `spacebot_auth_upsert_failures_total{reason}` (reason ∈ `invalid_principal_type | sqlx`) and `spacebot_auth_upsert_skipped_total` (fires only when `instance_pool` is unattached, a persistent non-zero is a startup-ordering regression). Test coverage: 12 integration tests in `tests/authz_data_model.rs` (all 5 migrations apply cleanly, user + team upsert idempotency, ownership round-trip, LegacyStatic rejection contract, None-for-missing, plus 5 raw-SQL tests — one per CHECK constraint). Design record: `docs/design-docs/entra-backfill-strategy.md` documents the no-auto-broadening policy for pre-existing resources and points at the Phase 10 orphan-sweep design. PR #101 also folded 11 safe Dependabot updates (clap 4.6.1, uuid 1.23.1, open 5.3.4, redb 4.1, tokio 1.52.1, tracing-appender 0.2.5, rmcp 1.5, desktop tracing-subscriber 0.3.23, docs fumadocs 16.8.1 + fumadocs-mdx 14.3.1). Phase 3 (Graph API client + group resolution) is next.

Three opt-in just recipes (`graphify-rebuild`, `graphify-query`, `graphify-clean`) wrap a `--directed`-preserving shell script. Not wired into CI or hooks. `.graphifyignore` at repo root mirrors `.gitignore` text-content entries plus graphify-specific exclusions (spacedrive/, vendor/, packages/api-client/src/schema.d.ts, graphify-out/ self-exclusion). Rationale + cost sizing + cross-project audit archived at `.scratchpad/completed/2026-04-21-graphify-research.md`. Complements the existing `code-review-graph` MCP (which handles structural Rust code queries); graphify's value is semantic cross-document clustering for design docs, RFCs, and screenshots. Landed 2026-04-21 alongside Phase 3 Entra work on `feat/entra-phase-3-graph-client`.

## Deployment
- Docker images published to GHCR (ghcr.io/jrmatherly/spacebot)
- Single binary, no server dependencies
- All data in embedded databases in a local data directory
- Release builds for Linux (x86_64, aarch64) and macOS (aarch64 only). Windows was dropped for `daemonize` Unix-only dep + bun-on-Windows preinstall gaps; Intel macOS dropped because `ort-sys` ships no prebuilt for `x86_64-apple-darwin`. Both exclusions are documented inline in `.github/workflows/release.yml`.
